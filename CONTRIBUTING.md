# Вклад в проект

## Принципы

1. **Модель — это контракт.** Любое изменение `src/gostforge/model/`
   затрагивает парсер, валидатор, экспортёр, фиксер и конструктор
   одновременно. Меняешь модель — следом обновляй `SCHEMA_VERSION`.

2. **Проверки изолированы.** Каждая проверка — отдельная функция,
   зарегистрированная через `@register("X.NN")` в модуле
   `src/gostforge/validator/checks/<category>.py`. Должна работать
   только с `Document` и `Profile`, ничего не зная о парсере, экспортёре,
   фиксере или конструкторе.

3. **Профили — это данные.** Логика не должна быть в YAML-файлах. Если
   правило слишком сложное для декларации — пишется как проверка-плагин,
   а в профиле остаётся только её конфигурация в `checks.X.NN.params`.

4. **Регрессионные тесты обязательны.** Любая новая проверка / эвристика
   парсера / новый фиксер сопровождается тестами. Тесты пишутся через
   прямое создание `Document` в коде, а не через .docx-фикстуры — это
   быстрее и стабильнее. Для интеграционных кейсов используется
   `tests/conftest.py::make_docx`.

5. **Безопасность фиксеров.** Фиксер мутирует Document, поэтому может
   применяться только к **смыслово-безопасным** правкам (двойные
   пробелы, точка в конце заголовка, NBSP). Любая правка, которая
   может изменить смысл текста — НЕ должна быть фиксером.

## Стиль кода

```bash
ruff check src tests
ruff format src tests
mypy src
python -m pytest -q
```

Все четыре команды должны проходить перед коммитом. **Важно:**
запускать pytest именно через `python -m pytest` (без модульного
запуска pytest не всегда видит editable-копию пакета).

## CI

`.github/workflows/ci.yml` запускает на каждом push / PR:

* **test** — `python -m pytest -q` на Python 3.11 и 3.12 (matrix).
* **lint** — `ruff check`, `ruff format --check`, `mypy --strict`.
  Сейчас в warn-only режиме из-за pre-existing baseline; новые
  изменения должны не ухудшать baseline (проверяется руками в PR).
* **docker** — `docker build` + `docker compose config` (валидация
  Dockerfile и compose-yaml; image публиковать не пытаемся).

Если CI красный из-за чужих pre-existing baseline-ошибок — это
нормально для warn-only шагов. Если упал именно pytest или docker
build — нужно чинить перед merge.

## Коммиты

Conventional Commits: `feat:`, `fix:`, `docs:`, `refactor:`, `test:`,
`chore:`. Тело коммита — на русском или английском, по вкусу.

Один коммит = одна логическая единица: одна проверка, один фиксер,
одно расширение парсера. **Не** объединять несколько проверок в один
коммит, даже если они логически связаны.

Примеры:
- `feat(check): H.06 — заголовок не висит внизу страницы`
- `feat(fixer): T.14 — авто-замена многократных пунктуаций (?!?! → ?!)`
- `fix(parser): корректная обработка вложенных таблиц`
- `feat(builder): метод .formula(latex) для добавления формулы`

## Как добавить новую проверку

1. Найди код проверки в `docs/checks-catalog.md`, проверь категорию.
2. Открой соответствующий модуль в `src/gostforge/validator/checks/`
   (или создай новый — название категории lower_case).
3. Зарегистрируй функцию через `@register("X.NN")`. Сигнатура:
   `def check_xxx(document: Document, profile: Profile) -> list[Violation]`.
4. Если у проверки есть параметры — читай их из `profile.checks["X.NN"].params`.
5. Если модуль новый — добавь его импорт в
   `src/gostforge/validator/checks/__init__.py`.
6. Включи проверку в `profiles/gost-7.32-2017.yaml` с дефолтными `params`.
7. Обнови статус в `docs/checks-catalog.md` (поставь ✓ в столбец «Статус»).
8. Тесты в `tests/test_check_<category>.py`: минимум 4 — `registered`,
   «всё хорошо» (нет violation), «нарушение найдено», краевой случай.

## Как добавить новый фиксер

1. Открой `src/gostforge/fixer/fixers/<category>.py`.
2. Зарегистрируй функцию через `@register("X.NN")`. Сигнатура:
   `def fix_xxx(document: Document, profile: Profile) -> list[FixApplied]`.
3. Функция **мутирует** Document на месте, возвращает список применённых
   правок (для отчёта и подсчёта).
4. Тесты в `tests/test_fixer.py`: minimum 3 — `registered`, fix applied
   (проверь что Document изменился), no-op (когда нечего исправлять).

## Как добавить шаблон для конструктора

1. Открой `src/gostforge/builder/templates.py`.
2. Реализуй функцию `<тип>_template(...) -> WorkBuilder`.
3. Шаблон должен создать WorkBuilder с обязательными разделами и
   плейсхолдерами так, чтобы итоговый Document проходил профильные
   проверки. Минимум — `S.01` (обязательные разделы).
4. Зарегистрируй в `gostforge/cli.py::new` (добавь в `click.Choice` и
   ветку диспетчера).
5. Тесты в `tests/test_builder_templates.py` — минимум 2: структура
   шаблона + интеграционный `save()` cycle.

## Этика проверок

Если правило требует **аннотировать чужой документ** (`gostforge
annotate`) — комментарии всегда конструктивные, с указанием
ГОСТа/пункта профиля. Запрещены: оценочные суждения о работе
(«плохо», «небрежно»), личные обращения, эмоциональная окраска.
Цель — помочь автору исправить, а не отчитать.

## Анонимизация фикстур

Если фикстура основана на реальной работе:
1. Снять метаданные через `gostforge.parser` → удалить ФИО / организацию.
2. Заменить ФИО, названия организаций, индивидуальные темы на синтетические.
3. Урезать до минимально необходимого фрагмента, демонстрирующего проблему.
4. Положить в `tests/fixtures/synthetic/` с описанием в комментарии.

Файлы из `tests/fixtures/real/` не должны попадать в git (запрещено
в `.gitignore`).
