# Использование Claude Code для разработки gostforge

Пошаговая инструкция: как с помощью Claude Code (CLI-агент от Anthropic) продолжить разработку проекта от текущей фазы 0 до полноценного MVP.

## 1. Установка и подготовка окружения

### Что понадобится

- **Node.js 18+** — для установки Claude Code.
- **Python 3.11+** — для самого проекта.
- **Git** — для работы с репозиторием.
- **Подписка Claude Pro / Max** — или API-ключ от console.anthropic.com.

Поддерживаемые ОС: macOS 13+, Ubuntu 20.04+/Debian 10+, Windows 10+ (включая WSL).

### Установка Claude Code

```bash
npm install -g @anthropic-ai/claude-code
```

Проверка:
```bash
claude --version
```

### Первый запуск и авторизация

```bash
cd path/to/gostforge
claude
```

При первом запуске Claude Code предложит авторизоваться. Самый простой путь — через OAuth с подпиской Pro/Max. Альтернатива — переменная окружения `ANTHROPIC_API_KEY` с ключом из консоли.

### Подготовка проекта

```bash
git clone https://github.com/AleksandrSotnikov/gostforge.git
cd gostforge

# Виртуальное окружение
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate

# Зависимости
pip install -e ".[dev]"

# Убедиться, что всё работает
pytest                              # должно быть 9 passed
gostforge profiles list             # → example-department, gost-7.32-2017
```

Если базовые тесты падают — **не запускай Claude Code, пока не починишь окружение**. Иначе агент будет пытаться чинить то, что сломано не им.

## 2. Как Claude Code узнаёт о проекте

В корне репозитория уже лежит файл `CLAUDE.md` — это контекст, который Claude Code автоматически читает в каждой сессии. Там описаны:
- архитектура и стек,
- ключевые правила (что можно / нельзя),
- структура папок,
- команды для тестов и линтеров,
- куда смотреть в `docs/`.

**Не нужно запускать `/init`** — `CLAUDE.md` уже подготовлен под этот проект. `/init` нужен только для пустых репозиториев, без контекста.

Дополнительно Claude Code будет читать любые файлы, на которые ты ссылаешься в промпте или которые он сам сочтёт релевантными.

## 3. Базовый рабочий цикл: Explore → Plan → Implement → Commit

Это рекомендованный паттерн работы с Claude Code, и он отлично ложится на наш проект.

**Explore.** Сначала проси изучить, а не писать. «Прочитай docs/architecture.md и docs/roadmap.md. Опиши, в какой фазе мы находимся и какая первая задача». Это даёт агенту контекст и проверяет, что он понимает проект.

**Plan.** Прежде чем писать код — нажми **Shift+Tab** для перехода в plan mode. В этом режиме Claude Code не редактирует файлы, а предлагает план. Прочитай план, поправь словами, согласуй — и только потом разрешай реализацию.

**Implement.** Когда план согласован — даёшь добро. Claude Code пишет код, запускает тесты, исправляет ошибки.

**Commit.** После каждой осмысленной единицы работы — коммит. Можно дать Claude Code это сделать: «Сделай коммит по conventional commits с описанием изменений». Но проверь diff перед `git push`.

## 4. Конкретные промпты для следующих шагов

Ниже — последовательность задач, соответствующая фазе 0–1 из `docs/roadmap.md`. Каждый промпт сформулирован так, чтобы Claude Code мог выполнить его за одну сессию.

### Шаг 1: Расширить парсер до извлечения метаданных

```
Прочитай docs/architecture.md и docs/page-sections.md. Текущая реализация
src/gostforge/parser/docx_parser.py — заглушка. Расширь её, чтобы извлекать
из .docx метаданные документа: title, author, supervisor (из docProps/core.xml).

Используй python-docx + lxml. Добавь синтетическую фикстуру в
tests/fixtures/synthetic/with_metadata.docx (создай её программно через
python-docx в conftest.py, не коммить готовый файл) и тест в
tests/test_parser.py, который проверяет извлечение метаданных. Не добавляй
ничего лишнего сверх метаданных. После реализации запусти pytest и mypy src.
```

### Шаг 2: Парсинг параграфов и базовой геометрии страницы

```
Расширь парсер: пусть он возвращает Document с одной PageSection, заполненной
геометрией страницы (margins_mm в миллиметрах из EMU, размер, ориентация)
и списком параграфов (Paragraph) с текстовым содержимым.

Создай синтетическую фикстуру с известными полями (например, 30/20/20/15 мм)
и текстом из 5 абзацев. Тест должен проверять, что после parse_docx():
- document.page_sections содержит ровно одну секцию,
- margins_mm совпадают с заданными в фикстуре с точностью до 0.5 мм,
- в секции есть нужное число параграфов с правильным текстом.

Не пытайся пока распознавать заголовки или стили — только plain paragraphs.
```

### Шаг 3: Реализовать проверки F.02, F.04, T.01, T.02, T.03

```
Сейчас в src/gostforge/validator/checks/formatting.py реализована только F.01.
Добавь проверки F.02 (формат бумаги A4), F.04 (положение номера страницы),
T.01 (шрифт Times New Roman), T.02 (кегль), T.03 (межстрочный интервал).

Описание проверок и их параметры — в docs/checks-catalog.md и в профиле
profiles/gost-7.32-2017.yaml. Для каждой проверки добавь по два теста
(положительный и негативный) в tests/test_validator.py или новый файл
tests/test_checks_formatting.py.

Перед коммитом запусти весь pytest и убедись, что mypy src проходит чисто.
```

### Шаг 4: Минимальный экспортёр

```
Реализуй src/gostforge/exporter/docx_exporter.py — функция export_docx должна
собирать минимальный .docx со следующим:
- поля страницы из profile.styles.page.margins_mm,
- шрифт и кегль из profile.styles.body,
- межстрочный интервал из profile.styles.body.line_spacing,
- параграфы из всех PageSection документа.

Колонтитулы, многосекционность, рисунки, таблицы — пока НЕ нужны. Это фаза 0.5,
минимум для round-trip-теста.

Добавь тест: создай Document вручную → export_docx → parse_docx → проверь,
что параметры и параграфы сохранились (round-trip).
```

### Шаг 5: Парсинг заголовков

```
Расширь парсер: распознавай заголовки и преобразуй их в LogicalSection.

Эвристика: заголовок — параграф со стилем Heading 1/2/3 ИЛИ
параграф с полужирным начертанием, расположенный отдельной строкой,
с outline_level в pPr.

Уровень определяется по стилю или по outline_level. Тело главы (параграфы
между заголовками) попадает в children этого LogicalSection.

Тесты: фикстура с тремя главами и подразделами, проверка структуры
document.page_sections[0].content.
```

### Шаг 6: Аннотирование .docx комментариями Word

```
Добавь подкоманду CLI: gostforge annotate <path> --profile <id> -o <out.docx>.
Она должна: парсить документ, прогонять проверки, для каждой violation
вставлять комментарий Word рядом с проблемным абзацем (или у начала
документа, если location не указывает на конкретный узел).

Используй python-docx + ручную работу с OOXML для вставки <w:commentReference>
и записи в word/comments.xml. Автор комментария — "gostforge", дата — текущая.

Текст комментария: «[F.01] Поле "top" = 25 мм, ожидается 20 мм».

Тест: создать документ с нарушением, прогнать annotate, открыть результат
парсером (или просто docx.Document) и убедиться, что комментарий присутствует.
```

### Шаг 7: Отчёт в xlsx

```
Реализуй опцию --report для gostforge check: при её указании результаты
прогона записываются в .xlsx через openpyxl.

Структура файла:
- Лист "Summary": одна строка на каждый проверенный файл, столбцы:
  файл, всего нарушений, errors, warnings, infos, итоговый статус (PASS/FAIL).
- Лист "Details": каждое нарушение отдельной строкой: файл, код, серьёзность,
  сообщение, локация, предложение.
- Лист "By check": агрегат по коду проверки: код, число файлов с этим
  нарушением, общее число вхождений.

Поддержи как один файл, так и папку (sorted glob *.docx).
```

## 5. Полезные команды Claude Code

| Команда | Что делает |
|---|---|
| `/help` | Список всех команд |
| `/context` | Показать, сколько контекста занято |
| `/clear` | Сбросить контекст текущей сессии |
| `/model` | Переключить модель (Sonnet / Opus / Haiku) |
| `Shift+Tab` | Plan mode: показывать план, не редактировать |
| `Esc` | Прервать текущее действие |
| `/compact` | Сжать историю, сохранив контекст |

**Когда переключать модель:** Sonnet — для большинства задач, Opus — для архитектурных решений и сложного дебага, Haiku — для рутины (форматирование, переименование, простые тесты).

## 6. Антипаттерны (как не надо)

**Слишком расплывчатые промпты.** «Допиши парсер» — плохо. «Расширь парсер, чтобы он извлекал поля страницы и список параграфов; добавь тест» — хорошо.

**Сразу прыгать в код без plan mode на сложных задачах.** Для всего, что трогает модель данных или несколько модулей сразу — сначала план, потом реализация.

**Запускать длинные сессии без `/clear` между несвязанными задачами.** Контекст копится, агент начинает «путаться». Закончил с парсером — `/clear`, потом берёмся за валидатор.

**Принимать diff не глядя.** Claude Code иногда чинит «соседнюю» проблему, которую ты не просил трогать. Каждый раз перед `git push` — `git diff --staged`.

**Коммитить реальные студенческие работы как фикстуры.** Это самое опасное в нашем проекте — у нас в `.gitignore` `tests/fixtures/*.docx`, но если агент создаст файл `tests/fixtures/Анисов_ПЗДП.docx`, его легко проглядеть. Все реальные работы → `tests/fixtures/real/` (под `.gitignore`), синтетические — генерируются программно в `conftest.py`.

**Просить «исправь все нарушения mypy/ruff» без разбора.** В лучшем случае агент добавит `# type: ignore` везде. Чини по одному и проверяй.

## 7. Чеклист перед каждым push

```bash
ruff check src tests       # ноль ошибок
ruff format --check src tests  # форматирование на месте
mypy src                   # типы чистые
pytest                     # все тесты зелёные
git diff --staged          # глазами просмотреть что коммитим
git log --oneline -5       # история выглядит осмысленно
```

Если что-то падает — не пушим. Claude Code умеет чинить по обратной связи: «pytest упал в test_xxx — посмотри, в чём дело, и поправь».

## 8. Куда двигаться дальше

После выполнения шагов 1–7 из раздела 4 — фаза 0 закрывается. Дальше по `docs/roadmap.md`:

- **Фаза 1.** Парсер до 80–90% покрытия, +15 проверок, отчёт xlsx, аннотирование. UI пока не нужен.
- **Фаза 2.** Конструктор UI (PyQt6 или Tauri+Web), колонтитулы в экспортёре, аннотирование Word-комментариями.
- **Фаза 3.** Формулы, библиография по 7.0.100, плагины проверок.

Каждый шаг — отдельная задача для Claude Code в формате выше: ссылка на доки, конкретный scope, обязательные тесты, проверки перед коммитом.
