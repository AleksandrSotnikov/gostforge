# Roadmap

## Фаза 0 — Фундамент (завершена) — 2–3 недели

Цель: на одной реальной работе пайплайн `parse → validate → report` сходится в работающий результат.

- [x] Структура проекта, документация
- [x] Schema модели документа (dataclass + SCHEMA_VERSION; Pydantic — для профиля)
- [x] Минимальный парсер `.docx → Model` (60–70% типовых работ): поля
      страницы, метаданные, параграфы с runs, заголовки, поле PAGE в footer
- [x] Минимальный экспортёр `Model → .docx`: поля страницы, стиль Normal,
      параграфы (TextRun → run с bold/italic), заголовки
- [x] Базовый профиль `gost-7.32-2017.yaml` (+ полноценный deep-merge при наследовании)
- [x] 5 ключевых проверок: F.01, F.04, T.01, T.02, S.01
- [x] CLI: `gostforge check` с юзер-френдли выводом (цвет, suggestion,
      Markdown-отчёт, --quiet), `gostforge profiles list`, `gostforge checks`
- [x] Базовые синтетические фикстуры (`tests/conftest.py::make_docx`)

Реальные .docx-фикстуры из архива (5+ анонимизированных работ) собираются
вместе с пилотной кафедрой в начале Фазы 1.

## Фаза 1 — MVP нормоконтроля — 4–6 недель

Цель: можно реально проверять поток работ.

- [x] Парсер расширен на основные блоки: поля страницы, метаданные,
      параграфы со стилями и runs, заголовки, таблицы и рисунки со
      склейкой подписей, поле PAGE в footer, `w:pgNumType`,
      `w:pageBreakBefore`, секция «Список использованных источников»
      → `BibliographyEntry`. Осталось на Фазу 2+: формулы (OMML),
      перекрёстные ссылки, реальные изображения.
- [x] **10/10 приоритетных проверок Фазы 1** реализованы.
- [x] **+15 проверок сверх плана**: F.02 (формат бумаги), F.03 (ориентация),
      F.05 (формат номера), T.07–T.11 (лингвистика), H.04 (последовательная
      нумерация), H.05 (иерархия), H.08 (точка в конце заголовка),
      S.02 (порядок разделов), S.07 (пустые разделы), I.03 (формат подписи
      рисунка), B.03 (формат подписи таблицы). **Итого 30 проверок** в реестре.
- [x] Команда `gostforge stats` — структурная статистика документа.
- [x] **Команда `gostforge fix` — автоисправление** безопасных нарушений
      (T.08, T.09, T.10, T.11, H.03, H.08) с `--only` и `--dry-run`.
      Реализована как симметричный движок `gostforge.fixer` с реестром
      по коду проверки.
- [x] Парсер читает header (симметрично footer) — поле PAGE в шапке
      распознаётся для F.04 в режиме `top_*`.
- [x] **Конструктор работ** (`gostforge.builder`): WorkBuilder / SectionBuilder
      с fluent API, шаблоны coursework / bachelor_thesis / research_report,
      команда `gostforge new`. Методы `.image()` (с реальной вставкой PNG/JPG),
      `.list(ordered=True|False)`. Парсер распознаёт списки.
- [x] **Аннотация .docx комментариями** (`gostforge annotate`): inline-маркеры
      `[CODE: message]` в проблемных параграфах исходного документа.
- [x] **+7 проверок**: H.02, T.12, T.13, I.05, I.06, B.08, B.09. **Итого 37
      проверок** в реестре.
- [x] Новый профиль `gost-r-2.105-2019.yaml` (ЕСКД, узкое правое поле).
- [x] Drag-and-drop UI на Streamlit: `gostforge ui` запускает локальный
      веб-интерфейс с загрузкой нескольких .docx, таблицей нарушений
      и скачиванием Markdown/Excel-отчётов.
- [x] Отчёт в формате `.xlsx` (плюс Markdown по тому же `--report`)
- [x] Адаптированный профиль кафедры (пример: `example-department.yaml`
      с кеглем 12, отступом 1.0 см, нумерацией с 4-й страницы, доп.
      разделом «Реферат», ослабленными параметрами R.04).
- [~] Регрессионный набор: ~16 синтетических фикстур через `make_docx`,
      покрывают все реализованные проверки. Цель — 20+ реальных
      анонимизированных работ от пилотной кафедры.

## Фаза 2 — Конструктор и расширение проверок — 6–10 недель

Цель: студент пишет работу в `gostforge`, преподаватель проверяет в `gostforge`.

- [ ] Студенческий UI с деревом блоков (PyQt6 или Tauri+Web)
- [ ] Базовые типы блоков: раздел, абзац, таблица, рисунок, список
- [ ] Экспорт `.docx` с корректными PageSection и колонтитулами
- [ ] Автоматические подписи и нумерация
- [ ] Валидатор расширяется до 40–50 проверок
- [ ] Менеджер профилей с наследованием
- [ ] Аннотирование `.docx` комментариями Word

- [ ] Формулы (LaTeX → OMML)
- [ ] Перекрёстные ссылки (внутренняя модель)

- [ ] Полный набор ~75 проверок
- [ ] Автоисправление безопасных нарушений
- [ ] Плагины проверок (Python)
- [ ] Шаблоны титульных листов

## Фаза 4 — Продакшен и масштаб — по мере спроса

- [ ] REST API
- [ ] Веб-версия (вынос UI в браузер)
- [ ] Командная работа: руководитель ↔ студент
- [ ] Интеграция с LMS (Moodle и др.)
- [ ] Маркетплейс профилей кафедр
- [ ] Мобильные клиенты для просмотра отчётов

## Принципы планирования

1. **Каждая фаза даёт рабочий продукт.** Не «недостроенная фаза 2», а «работает то, что обещано в фазе 1».
2. **Парсер — критический путь.** Без надёжного парсера всё остальное теоретическое.
3. **Профили — собираются на реальных кафедрах.** Не выдумываем «возможные» профили, добавляем по запросу.
4. **Регрессионные тесты обязательны.** Без 20+ фикстур к концу фазы 1 — стоп, доделываем фикстуры.
