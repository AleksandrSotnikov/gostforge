## 2026-Q2 — `gostforge diff --json` для CI/скриптов

К существующей CLI-команде `gostforge diff <a.docx> <b.docx>` добавлен
флаг `--json`: машиночитаемый структурированный вывод (totals, fixed,
introduced). Полезен для CI-гейтов («сравнить с предыдущей версией —
если стало хуже, упасть»), PR-ботов и автоматизации правок.

Структура JSON:

```json
{
  "profile_id": "gost-7.32-2017",
  "file_a": "...", "file_b": "...",
  "totals": {"a": 14, "b": 9, "errors_a": 10, "errors_b": 7},
  "fixed":       [{"code": "...", "severity": "...", "message": "...", "location": "..."}, ...],
  "introduced":  [...]
}
```

Exit code семантика та же: 1 если `errors_b > errors_a` (регресс),
иначе 0.

2 регрессионных теста в `tests/test_cli_diff.py`.

Итог: **1838+ тестов**, все четыре гейта чисты.

## 2026-Q2 — маркетплейс кафедральных профилей + прогресс-бар UI

### Маркетплейс кафедральных профилей (community-registry)

Roadmap-задача «Публичный реестр кафедральных профилей» переведена
в ✅ как локальный community-registry. YAML-профили в каталоге
`profiles/community/` — иллюстративные образцы кафедральных
профилей. Пользователь устанавливает выбранный в локальный реестр
одной кнопкой / командой.

**Структура:**
- `profiles/community/*.yaml` — каталог образцов. Пока 2 примера:
  `msu-vmk-coursework` (МГУ ВМК) и `spbstu-bachelor-thesis` (СПбПУ).
- `profile.list_community_profiles()` — список с метаданными
  (id, name, version, description, extends).
- `profile.read_community_profile_yaml(id)` — сырой YAML по id.

**CLI:**
- `gostforge profiles community list` — показывает доступные
  образцы с коротким описанием.
- `gostforge profiles community install <id> [--overwrite]` —
  устанавливает выбранный в локальную SQLite-БД (тот же механизм,
  что у обычного `profiles install`).

**UI:**
- Новая секция «Маркетплейс кафедральных профилей» на странице
  «Управление профилями». Поиск по названию/описанию + expander
  с описанием + кнопка «Установить» / «Переустановить» (если
  профиль уже в реестре).

## 2026-Q2 — SSE-прогресс в REST API

Roadmap-задача «WebSocket/SSE-уведомления в REST API (живой прогресс
длинной проверки)» переведена в ✅. Реализована через
Server-Sent Events (SSE) — проще WebSocket-а (HTTP, однонаправленно),
достаточно для прогресс-бара.

### Новый endpoint
- **`POST /check/stream`** — потоковая версия `/check`. Mime:
  `text/event-stream`. Заголовки против буферизации в reverse-proxy
  (`Cache-Control: no-cache`, `X-Accel-Buffering: no`).
- События в потоке:
  - `parse` — начали парсить .docx.
  - `check` — `{code, index, total}` перед запуском очередной
    проверки; клиент использует для процент-бара.
  - `done` — `{profile_id, violations, summary}`; гарантированно
    последнее событие.
  - `error` — фатальная ошибка парсинга.

### Реализация
- **Engine**: `validator.engine.validate_iter(document, profile)` —
  generator-вариант `validate`, yields `("check", code, index, total)`
  по очереди и `("done", violations)` в конце. Фильтр
  `disabled_checks` применяется в финале — семантика идентична
  `validate`.
- **API**: `_sse_event(name, payload)` хелпер формирует SSE-фрейм
  `event: <name>\\ndata: <json>\\n\\n`.
- **Submission в БД не пишется** — стрим рассчитан на интерактивные
  UI с прогресс-баром, а не на запись в историю. Для записи в
  историю используется обычный `/check`.

### Тесты — 6 новых
- `tests/test_api.py`: 5 SSE-тестов (happy path, прогресс-метаданные
  в check-событиях, summary в done, отказ на не-.docx, 404 на
  неизвестный профиль) + 1 unit на `validate_iter`.
- `_parse_sse_stream(text)` — хелпер парсинга SSE-потока (event +
  json payload).

### Документация
- `docs/api.md` §4.1.1: пример `curl -N` и JavaScript-клиент с
  чтением `ReadableStream` через `getReader()`.

Итог: **1828+ тестов**, все четыре гейта чисты.

## 2026-Q2 — drag-and-drop разделов и блоков

Roadmap-задача «Drag-and-drop порядка разделов и блоков» закрыта
полностью через `streamlit-sortables`. Реализована как toggle-able
β-функциональность: классические кнопки ↑/↓/⎘/✕ остаются основным
способом, DnD доступен для быстрого переупорядочивания мышкой.

### Реализация
- **Toggle «Drag-and-drop разделов»** в начале «Дерева разделов».
  Выключен по умолчанию — классические кнопки ↑/↓/⎘/✕ остаются
  стандартным способом.
- При включении: дерево заменяется на список с перетаскиванием
  мышкой (`streamlit_sortables.sort_items`). Кнопки управления
  разделом исчезают; активный раздел выбирается селект-боксом.
- **Стабильный матчинг**: каждой записи присваивается строка
  `"#{id}: {heading}"`, где `id` — стабильный per-section
  идентификатор. После reorder парсим id из префикса и
  переcтавляем `state.sections`. Если матчинг нестабилен
  (неизвестный id, нарушенный формат, расхождение количества) —
  порядок не меняется.
- **Активный раздел** автоматически отслеживает свою новую позицию
  после перетаскивания.
- **Зависимость**: `streamlit-sortables>=0.3.1` в `[ui]`-extra.
  При отсутствии toggle деградирует с warning.

### DnD блоков внутри раздела
- **Toggle «Drag-and-drop блоков»** в начале `_render_blocks_editor`.
  По умолчанию выкл — expander-ы с правкой остаются стандартом.
- При включении: блоки рендерятся как переcтавляемые метки
  «N. <тип / превью>». Чтобы редактировать — выключите toggle.
- Чистая функция `_reorder_blocks_by_dnd_items` симметрична
  секционной (`_reorder_sections_by_dnd_items`).

### Тесты — 10 новых (`tests/test_web_section_dnd.py`)
- 4 UI-смок через AppTest (toggle разделов, toggle блоков,
  замена кнопок селект-боксом, флаг в session_state).
- 6 unit на reorder-функции — реrder, защита от неизвестных id /
  расхождения количества / невалидного формата.

## 2026-Q2 — WYSIWYG-режим для параграфа (β)

Roadmap-задача «Inline-WYSIWYG для параграфа (требует
streamlit-quill)» переведена в ✅. У параграфа появился toggle
«WYSIWYG-режим (β)» — переключает per-run редактор на встроенный
Quill: пользователь печатает в реальном rich-text-поле, форматирует
через панель (B / I / U) или Ctrl+B/I/U, видит результат «как
в Word».

### Реализация
- **`_runs_to_html(runs)`** — text-run-ы сериализуются в `<p>`-блок
  с `<strong>` / `<em>` / `<u>`. Не-текстовые элементы (формулы /
  xref / citation) пропускаются — WYSIWYG-режим рассчитан на чисто
  текстовый параграф.
- **`_html_to_runs(html)`** — Quill HTML парсится через
  `html.parser.HTMLParser`. Стек bold / italic / underline учитывает
  вложенность; смежные run-ы с идентичным форматом склеиваются;
  хвостовой `<br>` от Quill стрипается.
- **UI**: новый toggle «WYSIWYG-режим (β)» в
  `_render_paragraph_inline_editor`. Доступен только если все
  элементы — text (формулы / ссылки / цитаты не поддерживаются).
  По умолчанию выключен — β-фича, классический per-run редактор
  остаётся стандартом.
- **Зависимости**: `streamlit-quill>=0.0.3` добавлен в `[ui]`-extra.
  При его отсутствии toggle деградирует с warning.

### Тесты — 13 новых (`tests/test_web_wysiwyg.py`)
- Конвертеры HTML↔runs (без Streamlit) — 12 unit-тестов:
  bold/italic/underline, escape, `<br>`, multi-paragraph, склейка,
  trailing-newline strip, round-trip, пропуск не-текстовых run-ов.
- UI-смок (через AppTest) — toggle отображается на параграфе.

Итог: **1812+ тестов**, все четыре гейта чисты.

## 2026-Q2 — per-section override схемы нумерации

Roadmap-задача «Per-section override схемы нумерации рисунков/таблиц»
переведена в ✅. У каждого раздела появилась опция переопределить
схему нумерации (`continuous` / `by_chapter`) только для этого
раздела, не трогая остальной документ.

### Реализация
- **WorkBuilder.numbering_override(figure=..., table=...)** —
  контекст-менеджер: временно меняет режим, после выхода
  восстанавливает. Сквозной `ordinal` (для xref) продолжает тикать
  независимо от режима — матчинг ссылок по позиции не ломается.
- **State**: новые поля `section["figure_numbering_override"]` и
  `section["table_numbering_override"]` (None / "by_chapter" /
  "continuous"). Невалидные значения молча игнорируются.
- **UI**: новый expander «Нумерация рисунков и таблиц в этом
  разделе» в редакторе активного раздела. Два selectbox-а:
  «По профилю» (default) / «Сквозная (1, 2, 3, …)» / «По главе
  (1.1, 1.2, 2.1, …)».
- **`_build_document_from_state`** оборачивает обработку каждой
  секции в `numbering_override`-контекст.

### Тесты
- `test_numbering_override_context_manager` — глобально continuous
  + override by_chapter для одной главы → правильные подписи.
- `test_numbering_override_separately_for_figures_and_tables` —
  фигуры/таблицы переопределяются независимо.
- `test_numbering_override_restores_previous_mode_on_exit` —
  после `with` режим восстанавливается.
- `test_build_document_from_state_applies_figure_numbering_override` —
  state→docx с override-ом не падает.
- `test_build_document_from_state_ignores_invalid_override` —
  невалидное значение игнорируется.
- `test_content_page_has_per_section_numbering_override_panel` —
  UI-смок: expander отображается.

## 2026-Q2 — итерация полировки веб-UI

Продолжение multi-page рефакторинга. Каждая страница получила
single-pattern **help-блок** `ℹ️ Что доступно на этой странице`

Регрессионные тесты гарантируют, что help-блок не исчезнет при
будущих правках.

### Help-блоки на 8 страницах активной функциональности

- **Структура / Содержимое / Проверка / Экспорт** — все 4 подстраницы
  Конструктора. На Содержимом дополнительно — палитра 6 типов
  блоков (+ Параграф / + Таблица / …) и inline-кнопки внутри
  параграфа (+ Текст / + Формула / + Ссылка / + Цитата).
- **Нормоконтроль** — описывает batch-загрузку, выбор профиля в
  sidebar, 4 вкладки результата (Проверка / Статистика /
  Автоисправление / PDF) и Markdown/Excel-отчёты внизу.
- **История** — submission-ы из локальной БД, фильтры в sidebar,
  две вкладки внутри (Нарушения / Обсуждение), управление статусом
  комментариев.
- **Редактор профиля** — поток работы: выбрать базовый профиль →
  9 вкладок параметров → сохранить. Объясняет наследование
  (`extends:`), почему профиль хранит только отличия от базового.
- **Управление профилями** — разделение между пользовательскими
  (SQLite-БД) и базовыми (вшитыми в пакет) профилями, импорт YAML
  с диска.

### Расширения Редактора профиля

- Новая вкладка **«Шаблон секций»** — read-only обзор
  `sections_template` (обязательные разделы работы), которое раньше
  было видимо только через YAML.
- Динамические виджеты для **`params` каждой проверки** (bool / int /
  float / str / list[str] / JSON). До этого params правили только
  редактируя YAML вручную.

### Прочие улучшения этого этапа

- **`gostforge stats --by-section --json`** — машиночитаемая
  разбивка статистики по разделам.
- **`gostforge doctor`** (с `--json`) — диагностика окружения
  (Python, ключевые зависимости, LibreOffice для PDF).
- **Расширенная статистика** — `logical_sections_level_2/_3`,
  списки/items, формулы, параграфы с inline-формулой / xref /
  citation, `bibliography_by_type`, `avg_words_per_paragraph`.
- **R.14** — конкретная диагностика формата DOI / URL (различает
  `htp://` vs `htps://`, отсутствие схемы, опечатки).
- **R.15** — опциональная HTTP-проверка достижимости URL (off
  по умолчанию; `timeout`, `max_urls` в params).
- **Live-превью многоуровневой шапки таблицы** в Конструкторе.
- **Content-based autosave** — debounce по sha256 хешу state,
  быстрее и без потери данных.
- **Три фикса конструктора по жалобе пользователя**:
  изображения переживают «Применить автофиксы», профиль не
  сбрасывается между подстраницами, «Продолжение таблицы N»
  при переносе таблицы через страницу включён по умолчанию.

## 2026-Q2 — многоагентный раунд: фиксеры U.*, парсер caption, стресс-тесты

Четыре независимых трека, выполненных параллельно через worktree-агентов:

### Track A — автофиксеры U.02 и U.03
- **U.02**: «10.кг» / «50,%» → «10 кг» / «50 %» (NBSP вместо знака
  препинания между числом и единицей).
- **U.03**: «10 кг.» → «10 кг» (стрипает завершающую точку у единиц СИ).
  Сохранены исключения проверки: «1990 г.» (≥1500 — это год) и
  «5 с.» (страница в библиографии) не трогаются.
- Структурный рефакторинг: U.* фиксеры переехали из `fixer/fixers/text.py`
  в отдельный модуль `fixer/fixers/units.py` (зеркально категории в
  валидаторе). U.01 теперь там же.
- Всего 28 автофиксеров (было 26).

### Track B — I.03/B.03 проверяют формат подписи из профиля
- До этого checks хардкодили шаблон «Рисунок N — Название». Теперь
  читают `profile.styles.{figure,table}.caption.format` и собирают
  регекс через общий хелпер `_build_caption_regex`. Поддерживается
  многоуровневая нумерация (1.2), приложенческий префикс (А.1),
  тире/дефис взаимозаменяемы.
- Back-compat: если в format-строке нет `{num}/{title}` —
  fallback на прежний хардкод. Параметр `allow_dot_after_number`
  сохранён.
- 10 новых тестов; сообщение и `details["expected_format"]` ссылаются
  на реальный формат из профиля.

### Track C — парсер извлекает Figure.number/Table.number
- Расширены `_FIGURE_CAPTION_TEXT_RE` / `_TABLE_CAPTION_TEXT_RE`: NBSP
  как whitespace (`[\s\xa0]+`), многоуровневая нумерация, приложение.
- Новая функция `_extract_caption_number()` устанавливает `.number` у
  Figure / Table при склейке подписи. До этого `.number` оставался
  None почти во всех форматах кроме дефолтного, что ломало
  downstream-логику (C.* cross-refs, I.05/B.05).
- 12 новых тестов на форматы: em-dash / period / colon / multilevel /
  appendix / NBSP / en-dash / hyphen + регресс «без подписи → None».

### Track D — стресс-тесты на документ 500+ страниц
- Фикстура ~140k слов через `builder.work` (30 разделов × 5 подразделов,
  40 параграфов в разделе, 10 рисунков, 8 таблиц).
- 4 теста с маркером `@pytest.mark.slow` (зарегистрирован в pyproject):
  validate, export_docx, parse_docx, compute_stats. Бюджеты выставлены
  ×3–5 от наблюдаемого времени (validate ~1.2 с, export ~2.3 с,
  parse ~9.8 с).

### Итог раунда

1687+ тестов (было 1654), 28 автофиксеров (было 26). Все четыре гейта
чисты (`ruff check`, `ruff format`, `mypy --strict`, `pytest`).

## 2026-Q2 — фокус на UX конструктора и обратная совместимость

### Парсер + экспортёр многоуровневой шапки таблицы
- Парсер распознаёт `<w:tblHeader/>` и раскладывает строки шапки по
  `Table.headers` (последняя) и `Table.extra_header_rows`
  (предыдущие — для «двойной/тройной» шапки по ГОСТ Р 2.105).
- Маркер «Продолжение таблицы N» автоматически фильтруется при
  парсинге — экспортёр добавит его обратно при необходимости.
- Координаты `merges` корректно сдвигаются.
- SCHEMA_VERSION → 0.4.0 (`Table.extra_header_rows`).

### Live-нормоконтроль: клик «→ К разделу»
- Каждое нарушение в live-панели конструктора получило кнопку
  «→ К разделу», которая активирует соответствующий top-level раздел.
- Маппинг `validator_section_id → state_index` строится через
  параллельный обход document.page_sections и state["sections"]
  (builder обходит state по порядку).
- `_iter_table_cells` теперь включает `extra_header_rows` — B.06
  корректно проверяет ячейки многоуровневой шапки.

### Конструктор: фичи нумерации, ширины и подписей
- **Высота рисунка**: `FigureStyleProfile.max_height_cm` (default 22 см).
  Экспортёр жмёт картинки и по высоте, не только по ширине, — подпись
  не съезжает на следующую страницу.
- **Схемы нумерации рисунков/таблиц**: `figure.numbering` /
  `table.numbering` (continuous | by_chapter). В приложениях всегда
  буквенно (А.1, Б.1) независимо от режима. WorkBuilder детектит
  «Приложение X» в heading и поддерживает per-chapter счётчики.
- **Повтор шапки таблицы** при переносе (`TableStyleProfile.repeat_header`,
  default True) — экспортёр ставит `<w:tblHeader/>` на header-row.
  Опционально `continuation_caption=True` — добавляет первой строкой
  «Продолжение таблицы N» (merged по колонкам, тоже tblHeader).
- **Многоуровневая шапка таблицы** (`Table.extra_header_rows`):
  «Группа 1 | Группа 2» сверху + «Подзаг A | B | C | D» снизу +
  данные. В web-редакторе поле «Доп. шапка» с авто-склеиванием
  пустых ячеек (`Группа 1||Группа 2|` → две группы по 2 колонки).
- **Профильный формат подписи** (`caption.format`) теперь респектится
  билдером — раньше формат хардкодился, и собственная же проверка
  I.05/B.05 ругалась на несоответствие.

### Багфиксы конструктора
- **Дублирование «Таблица N — Таблица N — ...»** при повторной вставке
  .docx: `_strip_caption_prefix` теперь распознаёт все формы
  разделителя (em-dash, en-dash, дефис, точка, двоеточие, NBSP,
  многоуровневую нумерацию) и применяется итеративно.
- **«Прилипание» состояния виджетов** при удалении/перемещении блока:
  стабильный per-block `id` (uuid4-hex) вместо позиционного индекса
  как ключ Streamlit-виджетов. Расширено на все уровни: блоки, run-ы,
  подразделы (level 2), пункты (level 3), top-level секции.
  Свежие id у новых сущностей теперь uuid4 (а не позиционные
  «sec-N», «sub-X-Y», которые могли пересечься после удаления).

### Автофиксеры формата и геометрии страницы
- **F.01/F.02/F.03** — поля → профиль (допуск 0.5 мм), формат бумаги
  → A4, ориентация → portrait (приложения не трогаются).
- **H.01/H.02** — формат заголовков 1–2 уровня (шрифт/кегль/жирность/
  цвет — лечит «синий Cambria» из шаблона Word). Регистр и текст
  не меняются.
- **T.01/T.02/U.01** — шрифт и кегль основного текста, неразрывный
  пробел перед единицей СИ.
- Все правят только ЯВНЫЕ run-атрибуты, наследуемые значения
  не трогают, переиспользуют предикаты соответствующих проверок.
- **Итог:** 26 автофиксеров (было 18).

### UI «Автоисправление»: детализация правок
- Вкладка показывает, **что именно** исправлено: сводка по кодам с
  количеством (`F.01 ×3, T.08 ×5`) и раскрывающийся список

- Шапка вкладки динамическая (число доступных фиксеров + актуальный
