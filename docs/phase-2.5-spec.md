# Фаза 2.5 — Пословное редактирование в визуальном конструкторе

> **Статус:** ТЗ. Реализация ещё не начата.
> **Предшественник:** Фаза 2 (визуальный конструктор на Streamlit, абзац = `text_area`).
> **Преемник:** Фаза 3 (REST API, веб-сервис, маркетплейс профилей).

## 1. Контекст и обоснование

В текущей реализации визуального конструктора
(`src/gostforge/web/builder_editor.py`) каждый параграф редактируется
как одна цельная многострочная ячейка `st.text_area`. Это даёт быстрый
старт, но накладывает три принципиальных ограничения:

1. **Нет локального форматирования внутри абзаца.** Студент не может
   выделить отдельное слово курсивом или жирным — а ГОСТ 7.32 этого
   требует во множестве мест (термины при первом упоминании, латинские
   названия видов, обозначения переменных, акценты).
2. **Нет inline-элементов.** Формула посреди предложения, перекрёстная
   ссылка «(см. рисунок 3)», цитата «[5, с. 12]» — всё это сейчас
   приходится вписывать руками в виде сырого текста, который ломает
   проверки C.* (перекрёстные ссылки) и R.* (цитирования) и не
   связывается с реальными объектами модели.
3. **Нет атомарных правок.** Любое изменение параграфа триггерит
   полную перерисовку всего блока в Streamlit и теряет позицию курсора;
   undo/redo работает только на уровне строки браузера, а не модели.

Модель документа уже готова к мелкозернистому контенту:
`Paragraph.content: list[InlineElement]`, где `InlineElement = TextRun
| CrossRef`. Пословное редактирование — это, по сути, поднятие этой
структуры в UI без миграции данных.

## 2. Цель и определение готовности

**Цель.** Перевести редактирование контента в визуальном конструкторе
с уровня «целый абзац строкой» на уровень **inline-элементов**:
форматированный фрагмент, перекрёстная ссылка, inline-формула,
библиографическая цитата.

**Определение готовности (Definition of Done).**

1. В UI каждый абзац представлен последовательностью именованных
   inline-элементов. Студент видит и редактирует их по-отдельности.
2. Поддержаны 4 типа inline-элементов: `TextRun`, `CrossRef`,
   `InlineFormula`, `Citation` — все имеют визуальный редактор и
   round-trip сериализацию в JSON-state.
3. Round-trip state → `Document` → `.docx` → парс → state не теряет
   ни одного inline-элемента и его атрибутов на синтетических
   фикстурах (новый набор тестов `tests/test_inline_*.py`).
4. Существующие 746+ тестов остаются зелёными; mypy --strict не
   деградирует; ruff чист.
5. Конструктор-генерируемый документ по-прежнему проходит ≥29 из
   30 проверок профиля `gost-7.32-2017` (без регрессий относительно
   Фазы 2).
6. Документация: `docs/builder.md` дополнен разделом «4.1 Пословное
   редактирование», `docs/architecture.md` — описанием расширенной
   модели inline-контента, README — упоминанием в разделе «Визуальный
   редактор».

## 3. Не-цели (что НЕ войдёт в Фазу 2.5)

Эти направления намеренно вынесены, чтобы не раздуть фазу:

- **Совместное редактирование** (multiplayer / OT / CRDT).
  Конструктор остаётся однопользовательским локальным инструментом.
- **История версий за пределами JSON-save.** Undo/redo — да; git-like
  диффы — нет.
- **WYSIWYG-предпросмотр финального .docx.** Превью остаётся
  структурным («дерево» разделов и блоков), не пиксельным.
- **Полнотекстовый rich-text-редактор уровня Word.** Никаких таблиц
  внутри абзаца, плавающих изображений с обтеканием, многоколоночной
  вёрстки и т. п. — это контент-структура, а не вёрстка.
- **Миграция данных пользовательских JSON-save из Фазы 2.** Их формат
  расширяется обратно совместимо (см. §6.3), но автоконвертация не
  выполняется — старый save просто откроется, новые поля будут
  пустыми списками.
- **Голосовой ввод / AI-подсказки.** Отдельная фаза при необходимости.

## 4. Архитектура

### 4.1. Модель данных

`src/gostforge/model/__init__.py` расширяется новыми вариантами
`InlineElement`. **SCHEMA_VERSION → 0.3.0.** Все новые поля имеют
default или Optional, поэтому миграция парсера не требуется — но
проверки и экспортёр должны корректно обрабатывать новые типы.

```python
@dataclass
class InlineFormula:
    """LaTeX-формула, отрисовываемая в потоке текста.

    На экспорте → OMML внутри run параграфа (а не отдельный <m:oMathPara>).
    На парсинге → распознаётся по <m:oMath>, лежащему внутри <w:r>.
    """
    latex: str
    # id опционален: inline-формулы обычно не нумеруются.
    id: str | None = None


@dataclass
class Citation:
    """Inline-цитата на источник из Document.bibliography.

    Отображается как «[N]» или «[N, с. P]» (по заданному template).
    Номер N вычисляется на экспорте из позиции источника в
    Document.bibliography (1-based).
    """
    source_id: str            # ссылка на BibliographyEntry.id
    pages: str | None = None  # "12", "12-15", "12, 17-20"
    template: str = "[{n}]"   # либо "[{n}, с. {pages}]"


# CrossRef уже существует, но дополним:
@dataclass
class CrossRef:
    target_id: str
    display_template: str = "{kind} {num}"
    # NEW: явный текст «(см. ...)», «согласно ...» — добавляется
    # перед автогенерируемой ссылкой. None = только сам номер.
    prefix: str | None = None


InlineElement = TextRun | CrossRef | InlineFormula | Citation
```

`TextRun` уже имеет `bold`, `italic`, `superscript`, `subscript`,
`font`, `size_pt` — этого достаточно. Дополнительно вводим
`underline: bool | None = None` (используется в кафедральных
методичках для названий теорем) и `color_hex: str | None = None`
(только для синтаксической подсветки кода в `BlockType.CODE`; в
обычном тексте проверка X.* запретит непустой `color_hex`).

### 4.2. State в Streamlit

Расширение схемы `st.session_state.builder_state`. Параграф вместо

```json
{"kind": "paragraph", "text": "Привет мир"}
```

становится

```json
{
  "kind": "paragraph",
  "runs": [
    {"kind": "text", "text": "Привет ", "bold": false, "italic": false},
    {"kind": "text", "text": "мир", "bold": true, "italic": false},
    {"kind": "xref", "target_id": "fig-1", "prefix": " (см. "},
    {"kind": "text", "text": ")"}
  ]
}
```

Поле `"text"` для обратной совместимости остаётся валидным: при
загрузке старого save функция `_normalize_paragraph_state()`
конвертирует `text` → один `TextRun` без форматирования.

### 4.3. UI-компоненты

Streamlit не имеет нативного rich-text-редактора с
inline-элементами. Рассматриваем два варианта:

| Подход | Плюсы | Минусы | Решение |
| --- | --- | --- | --- |
| **A. Нативный Streamlit: список run-ов как `st.text_input` + чекбоксы B/I/U + отдельная панель «вставить inline-элемент».** | Без зависимостей, мгновенно работает, легко тестируется, не ломает session_state-архитектуру. | Менее «приятный» UX: каждый run — отдельная строка ввода. | **Принят.** Это MVP Фазы 2.5. |
| B. Кастомный `streamlit.components.v1.html` с TipTap / ProseMirror и JSON-bridge. | Настоящий WYSIWYG. | Большая зависимость (npm-сборка), сложно тестировать, дублирует логику модели в TS. | Откладываем до Фазы 3. |

**MVP-редактор параграфа** (вариант A):

```
┌────────────────────────────────────────────────────────────────┐
│ Run 1: [Привет ▌      ] [B] [I] [U] [a²] [a₂] [×]              │
│ Run 2: [мир ▌         ] [B*] [I] [U] [a²] [a₂] [×]             │
│ ✦ Ссылка: → рисунок «Архитектура» (fig-1)    [✎] [×]           │
│ Run 3: [). Подробнее...] [B] [I] [U] [a²] [a₂] [×]             │
│                                                                │
│ ┌──────────────────────────────────────────────────────────┐   │
│ │ + Run  │ + Перекр. ссылка  │ + Формула  │ + Цитата       │   │
│ └──────────────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────────────┘
```

- `+ Run` добавляет пустой `TextRun` в конец списка.
- `+ Перекр. ссылка` открывает `st.selectbox` со списком всех
  рисунков/таблиц/формул из текущего state (с их `caption` для
  выбора), позволяет указать prefix.
- `+ Формула` — поле `LaTeX` + live-preview через `st.latex()`.
- `+ Цитата` — `st.selectbox` со списком записей библиографии и
  опциональным полем `pages`.
- `[×]` рядом с run — удалить; перетаскивание не реализуем (Streamlit
  не поддерживает HTML5 DnD), вместо него — кнопки `↑` / `↓`.

### 4.4. Сериализация и обратная совместимость

В `builder_editor.py` две парные функции расширяются:

- `_build_document_from_state(state)` — теперь обходит `runs`
  параграфа и собирает `list[InlineElement]` через factory-функции
  `_state_to_run(item)`, выбирая ветку по `kind` (`text` / `xref`
  / `formula` / `citation`).
- `_document_to_state(document)` — обратная конвертация. Параграф
  получает `runs: list[dict]`, поле `text` больше не пишется
  (но при загрузке старого save читается, см. ниже).

`_normalize_paragraph_state(p: dict) -> dict` — нормализатор,
вызываемый при загрузке JSON: если у параграфа есть `text` и нет
`runs`, создаёт `runs = [{"kind": "text", "text": p["text"]}]` и
удаляет `text`. Это гарантирует, что save файлы Фазы 2 открываются.

`WorkBuilder` получает новый низкоуровневый метод

```python
SectionBuilder.rich_paragraph(elements: list[InlineElement]) -> SectionBuilder
```

рядом с существующим `.paragraph(text: str)`. Существующий
`.paragraph(text)` остаётся (для CLI-шаблонов и для unit-тестов
программного API), но внутри сводится к `.rich_paragraph([TextRun(text=text)])`.

### 4.5. Парсер и экспортёр

- **Экспортёр** (`docx_exporter.py`): расширить `_write_paragraph`
  обработкой каждого варианта `InlineElement`:
  - `TextRun` → текущая логика (`run.bold`, `run.italic`, ...).
  - `CrossRef` → пара `<w:fldSimple w:instr=" REF target_id "/>`
    + видимый текст из template; при отсутствии target — fallback
    в `«?»` + warning в лог.
  - `InlineFormula` → `<w:r><m:oMath>...</m:oMath></w:r>`
    через `latex_to_omml()` (уже есть в экспортёре для блочных
    формул).
  - `Citation` → текстовый run «[N]» / «[N, с. P]», где N
    вычисляется как 1-based индекс `Document.bibliography`.
- **Парсер** (`docx_parser.py`): распознавание тех же конструкций
  при чтении документа. Для Фазы 2.5 это уже частично работает
  (CrossRef через `<w:fldSimple>`, формулы через `<m:oMath>`); нужно
  доработать распознавание цитат — для них на парсере вводим
  эвристику «текст вида `[N]` или `[N, с. P]` рядом с библиографией
  → `Citation`, иначе оставляем как `TextRun`».

### 4.6. Undo/Redo и auto-save

- **Undo/Redo** реализуем на уровне `builder_state` через
  кольцевой буфер из 50 последних snapshot'ов:
  `st.session_state.builder_history: list[dict]` +
  `builder_history_cursor: int`. Перед каждой мутацией state
  делаем `copy.deepcopy(state)` и пушим в буфер. Кнопки `Ctrl+Z`
  / `Ctrl+Y` (через `st.button` в sidebar, поскольку Streamlit
  не имеет глобальных hotkey API).
- **Auto-save** в session_state уже работает (любая мутация
  state → перерисовка). Добавляем периодический dump в
  `~/.gostforge/autosave/<session_id>.json` каждые 30 секунд
  (через `st.cache_data(ttl=30)` маркер + `st.experimental_rerun()`
  на тике). При запуске UI проверяется наличие autosave и
  предлагается восстановить.

## 5. Сценарии использования (acceptance scenarios)

### S1. Жирное слово в середине предложения

1. Студент открывает параграф «Целью работы является разработка системы.»
2. Дважды нажимает `+ Run` чтобы разбить параграф (либо
   автоматический сплит по выделению — отложено в not-goals).
3. Получает 3 run-а: `"Целью работы является разработка "`,
   `"системы"`, `"."`. На втором ставит `[B]`.
4. Генерирует `.docx`. Открыв в Word, видит слово «системы»
   полужирным.
5. Парсит обратно — `_document_to_state` восстанавливает три run-а
   с правильным `bold` на втором.

### S2. Перекрёстная ссылка на рисунок

1. У студента в state есть рисунок с `caption = "Архитектура"` и
   `id = "fig-1"`.
2. В параграфе нажимает `+ Перекр. ссылка`, выбирает «Архитектура (fig-1)»,
   указывает `prefix = " (см. "`, добавляет следующий run `")."`.
3. В сгенерированном `.docx` появляется поле `REF fig-1` →
   автоматически отображается «(см. рисунок 1).»
4. Проверка C.01 проходит (ссылка валидна), C.02 проходит (на
   рисунок есть хотя бы одна ссылка).

### S3. Inline-формула

1. В параграфе «Энергия фотона E = h·ν где h — постоянная Планка.»
2. После «E = » добавляется `+ Формула` с LaTeX `h\nu`.
3. Live-preview через `st.latex("h\\nu")` показывает рендер.
4. В `.docx` экспортируется OMML inline (внутри run, а не как
   отдельный абзац).

### S4. Цитата с указанием страницы

1. У студента в библиографии 5 записей.
2. Параграф «Это подтверждается в …»; нажимает `+ Цитата`, выбирает
   запись «Иванов И. И., 2023», pages = `"12-15"`.
3. На экспорте получает «Это подтверждается в [3, с. 12-15].»
4. Проверка R.04 (формат ссылок) проходит.

### S5. Совместимость со старыми save

1. Студент открывает JSON, сохранённый в Фазе 2 (с полем `text`).
2. `_normalize_paragraph_state` тихо конвертирует в формат `runs`.
3. UI показывает один `TextRun` per параграф; ничего не сломано.

## 6. Тесты

Новые модули тестов:

- `tests/test_inline_model.py` — конструирование всех 4 типов
  inline-элементов; сериализация в dict и обратно.
- `tests/test_inline_state_roundtrip.py` —
  `_document_to_state(_build_document_from_state(state))` идемпотентно
  на 10+ фикстурах (только текст, текст+CrossRef, текст+формула,
  текст+цитата, всё вместе, вложенные элементы в разных порядках).
- `tests/test_inline_exporter.py` — после
  `export_docx(doc) → parse_docx(...)` сохраняются все inline-элементы
  включая bold/italic/underline на нужных позициях.
- `tests/test_inline_parser.py` — парсер распознаёт inline-формулы
  из OMML внутри `<w:r>` и не разбивает их на TextRun-ы.
- `tests/test_inline_compat.py` — `_normalize_paragraph_state`
  правильно обрабатывает JSON формата Фазы 2.
- `tests/test_builder_rich_paragraph.py` — программный API
  `SectionBuilder.rich_paragraph(...)` работает и сохраняется.
- `tests/test_inline_undo.py` — undo/redo на 10 шагах не теряет
  state, обращает мутации.

Минимум **40 новых тестов**, общая база — **≥ 786 тестов**, все
зелёные на `python -m pytest -q`.

## 8. Риски и митигация

| Риск | Митигация |
| --- | --- |
| Streamlit не умеет fokus/выделение текста — нельзя сделать «выделил → нажал B». | Принимаем: студент сначала делит абзац на run-ы, потом форматирует. В UI добавляем подсказку. |
| `deepcopy` на каждой мутации — медленно при больших state. | Профилируем; если >100мс на типичном документе, переходим на patch-based history (jsondiff). |
| Существующие проверки (C.*, R.*) могут не учесть, что текст ссылки теперь поле REF. | Парсер уже извлекает текст из `<w:fldSimple>`; добавляем тест per check, чтобы убедиться что C.01/C.02/R.04 продолжают работать. |
| Цитаты-эвристика (распознавание `[N]` в тексте) даёт false-positives на математических индексах вида `[0; 1]`. | Эвристика срабатывает только если N — целое число от 1 до `len(bibliography)`. Иначе остаётся `TextRun`. |
| `bytes`-передача файлов изображений во временные файлы при auto-save → ссылка ломается между сессиями. | Auto-save кладёт изображения base64 в JSON; при restore — пишет обратно в `tempfile`. |

## 9. Стратегия использования агентов

После §4.1 (расширение модели — основная сессия) можно запустить
параллельно через `isolation: worktree`:

- **A1**: экспортёр inline-элементов (§4.5, экспортёр) + round-trip
  тесты.
- **A2**: парсер inline-формул и citations (§4.5, парсер) + тесты.
- **A3**: UI inline-редактор параграфа (§4.3) + compat-тесты
  `_normalize_paragraph_state`.
- **A4**: undo/redo + auto-save (§4.6) + тесты.

Все агенты получают на вход эту спецификацию. Интегратор (основная
сессия) после каждого агента:
