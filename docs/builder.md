# Конструктор работ

Модуль `gostforge.builder` — высокоуровневый fluent-API над моделью документа.
Студент описывает работу декларативно: разделы, параграфы, таблицы, рисунки,
список литературы. На выходе — `.docx`, который сразу проходит профильные
проверки нормоконтроля.

При сохранении конструктор автоматически:

- ставит правильные поля страницы, шрифт и межстрочный интервал (по профилю);
- проставляет `page_break_before=True` у первого параграфа каждого раздела
  верхнего уровня (кроме самого первого);
- добавляет футер с полем `PAGE` (номер страницы);
- ставит `pgNumType.start = 3`;
- собирает список литературы в `Document.bibliography`.

## Веб-интерфейс (`gostforge ui`)

Команда

```bash
gostforge ui
```

запускает Streamlit-веб-приложение в браузере (десктоп; интерфейс не
адаптирован под мобильные). Навигация — sidebar слева, страницы
сгруппированы по workflow:

```
🏠 Старт
  └ Главная                      (?page=home)
🔍 Работа с документом
  └ Нормоконтроль                (?page=normocontrol)
🛠️ Конструктор
  ├ 🏗️ Структура                (?page=builder-structure)
  ├ ✏️ Содержимое               (?page=builder-content)
  ├ ✅ Проверка                  (?page=builder-validation)
  └ 📤 Экспорт                   (?page=builder-export)
⚙️ Настройка
  ├ Редактор профиля             (?page=profile-editor)
  └ Управление профилями         (?page=profile-manager)
📚 Справка
  ├ История                      (?page=history)
  └ Документация                 (?page=docs)
```

URL отражает страницу: ссылку на конкретный режим можно отправить
руководителю, refresh её не сбрасывает, браузерная история работает.
На старте Streamlit lazy-загружает только активную страницу.

Ниже описан workflow каждой страницы. Детали внутреннего устройства
конструктора (state, undo/redo, inline-редактирование) вынесены в
раздел [«4. Визуальный редактор»](#4-визуальный-редактор-streamlit-webapp).

### Главная

Стартовый экран-обзор:

- краткое описание возможностей системы и блок **«С чего начать»**;
- шпаргалка по ключевым требованиям **ГОСТ 7.32-2017**;
- метрики: число доступных проверок и число профилей.

Отсюда удобно перейти в нужный режим через верхний переключатель.

### Нормоконтроль

Проверка готового документа без редактирования:

1. Загрузить `.docx` (uploader).
2. Выбрать профиль ГОСТ.
3. Запустить проверку — получить **цветной отчёт** с группировкой
   нарушений по уровню важности.
4. Выгрузить отчёт в **Excel** или **Markdown**.
5. При необходимости — **применить автофиксы** к найденным нарушениям.

### Конструктор

Полноценная сборка работы «с нуля» или на основе импортированной `.docx`.
Разнесён на **4 подстраницы**, отражающие workflow:

#### Структура (`?page=builder-structure`)

Управление скелетом работы:

- **Sidebar «Раздел из шаблона»**. Выпадающий список готовых разделов:
  Введение, Заключение, Реферат, Содержание, Список использованных
  источников, Приложение, Глава, **Титульный лист (вставить вручную)**.
- **Кнопка «Собрать каркас по ГОСТ»** наполняет пустой документ полным
  структурным набором разделов сразу.
- **Bulk-операции**: удалить пустые параграфы, привести заголовки к
  Title Case, авто-нумерация глав со снятием нумерации со структурных
  разделов.

Особенности структуры:

- **Приложения** авто-нумеруются буквами по ГОСТ: А, Б, В, … —
  без букв Ё, З, Й, О, Ч, Ь, Ы, Ъ.
- **Титульный лист** — место под ручную вставку; нормоконтролем он
  не проверяется.

Здесь же видны **«Прогресс работы»**, **«Готовность работы»**
(чек-лист обязательных элементов с навигацией) и **live-preview
оглавления**.

#### Содержимое (`?page=builder-content`)

Редактирование активного раздела:

- параграфы, таблицы, рисунки, списки, формулы и оглавление (TOC);
- **inline-редактирование** на уровне TextRun-ов (жирный/курсив/sup/sub,
  inline-формулы, перекрёстные ссылки, цитаты);
- управление **подразделами 2–3 уровня**;
- **undo / redo**;
- **поиск-замена**.

Над редактором — **breadcrumb**: «📍 Раздел N / M: «название» · блоков
X · подразделов Y». Можно быстро переключаться на другой раздел через
dropdown в начале страницы.

**Изображения** вшиваются прямо в state как data-URI — они переживают
сохранение в `.json` и открытие в новой сессии.

#### Проверка (`?page=builder-validation`)

- **Live-нормоконтроль** текущего state с детальным списком нарушений
  (код + сообщение + подсказка + кнопка **«→ К разделу»**, которая
  переключает на нужный раздел в «Содержимом»).
- **Импортированные нарушения** и **комментарии рецензента** из
  `.docx`, если работа была загружена.

#### Экспорт (`?page=builder-export`)

- скачать **.docx** (primary-кнопка);
- получить превью и скачать **.pdf**;
- скачать **.md**;
- скачать **.html**;
- опция **«Применить автофиксы перед скачиванием»**.

**Импорт и сохранение state** — в общем sidebar Конструктора
(виден на всех 4 подстраницах): можно загрузить готовую `.docx` и
разложить её в конструктор для дальнейшей правки, а также сохранить
рабочее состояние как `.json`.

#### Типичный сценарий

1. **Структура** — нажать «Собрать каркас по ГОСТ» (или добавить
   разделы из шаблонов по одному); расставить порядок.
2. **Содержимое** — заполнить разделы: параграфы, таблицы, рисунки,
   списки, формулы.
3. **Проверка** — посмотреть live-нормоконтроль, исправить ошибки
   (клик «→ К разделу» возвращает на «Содержимое»).
4. **Экспорт** — скачать в нужном формате с автофиксами.

### Редактор профиля

Визуальная правка параметров одного профиля без ручного редактирования YAML:

- параметры оформления: **страница**, **основной текст**,
  **заголовки 1–4 уровней**, **таблицы**, **рисунки**, **списки**;
- **состав и важность проверок**;
- **метаданные** профиля.

Сохранить профиль можно как **полный снимок** или как **наследник**
(через `extends`). Список установленных и удаление — на отдельной
странице «Управление профилями».

### Управление профилями

Каталог пользовательских профилей в локальной БД:

- список установленных профилей (ID / название / версия / дата);
- **удаление** профиля;
- **загрузка готового YAML с диска** (drag-and-drop) — полезно, когда
  кафедра прислала готовый профиль файлом.

### История

Журнал прошлых проверок из локальной БД:

- ранее выполненные проверки (submission-ы);
- **обсуждение руководитель ↔ студент** в виде комментариев.

### Документация

Встроенный просмотр руководства прямо в интерфейсе — без перехода
во внешние источники.

## 1. Минимальный документ

```python
from gostforge.builder import work

(
    work("Курсовая по нормоконтролю", author="Иванов И. И.", year=2026)
    .section("Введение")
        .paragraph("Актуальность темы заключается в ...")
        .paragraph("Целью работы является ...")
    .section("Глава 1. Анализ предметной области")
        .subsection("1.1 Обзор существующих решений")
            .paragraph("В настоящее время существует множество подходов ...")
        .subsection("1.2 Постановка задачи")
            .paragraph("Сформулируем задачу следующим образом ...")
    .section("Заключение")
        .paragraph("В ходе работы было ...")
    .section("Список использованных источников")
        .reference("Иванов И. И. Программирование. — М. : Наука, 2023. — 320 с.")
    .save("coursework.docx", profile="gost-7.32-2017")
)
```

`save()` сам прогоняет результат через валидатор и поднимает `ValueError`,
если есть error-уровневые нарушения. Если нужна только модель (без записи
файла) — используйте `.build()`:

```python
builder = work("Курсовая").section("Введение").paragraph("Текст")
document = builder.build()   # gostforge.model.Document
```

## 2. Шаблоны

В `gostforge.builder.templates` лежат скелеты с уже добавленными
обязательными разделами и плейсхолдером `<Заполните этот раздел>`:

```python
from gostforge.builder.templates import (
    coursework_template,
    bachelor_thesis_template,
    research_report_template,
)

builder = coursework_template(
    title="Курсовая по нормоконтролю",
    author="Иванов И. И.",
    year=2026,
)
builder.save("coursework.docx", profile="gost-7.32-2017")
```

Шаблоны возвращают `WorkBuilder`, в который можно продолжать добавлять
разделы/параграфы перед `save()`.

## 3. CLI-команды конструктора

Полный набор CLI для работы со state-файлами и .docx:

### 3.1 Создание

```bash
# .docx напрямую (с плейсхолдерами для заполнения)
gostforge new my-coursework.docx \
    --template coursework \
    --title "Курсовая по нормоконтролю" \
    --author "Иванов И. И." \
    --year 2026

# JSON-state для дальнейшего редактирования в UI или скриптом
gostforge new-state \
    --template coursework \
    --title "Анализ" \
    --author "Иванов И. И." \
    --year 2026 \
    -o draft.json
```

Доступные шаблоны: `coursework`, `bachelor_thesis`, `research_report`, `empty`
(минимальный каркас с Введением и Списком источников).

### 3.2 Импорт / экспорт

```bash
# Разложить готовую .docx в JSON-state для редактирования
gostforge import-docx work.docx -o state.json

# Извлечь структуру PDF в JSON-state (форматирование не переносится,
# нужен extra [import-formats]: pip install "gostforge[import-formats]")
gostforge import-pdf work.pdf -o state.json

# Собрать .docx из JSON-state (зеркало import-docx)
gostforge generate state.json -o new.docx
gostforge generate state.json -o new.docx --profile gost-r-2.105-2019

# Markdown (GFM) — для git-ревью, Obsidian, конвертации через pandoc
gostforge export-md state.json -o draft.md
gostforge import-md draft.md -o state.json
```

`export-md`/`import-md` поддерживают rich-форматирование: `**bold**`,
`*italic*`, `***bold+italic***`, GFM-таблицы, формулы `$$ latex $$`,
изображения `![caption](path)`.

### 3.3 Правки и сравнение

```bash
# Применить автофиксы к state-файлу (без UI)
gostforge apply-fixes draft.json -o fixed.json
gostforge apply-fixes draft.json -o fixed.json --only T.08,T.14

# Сравнить две версии state
gostforge diff-state old.json new.json              # summary
gostforge diff-state old.json new.json --mode unified  # построчный diff JSON
```

`diff-state` summary показывает: добавленные/удалённые/изменённые разделы,
изменение блоков/подразделов/disabled_checks/источников. Удобно для аудита
автофиксов и code-review.

### 3.4 Полный цикл

```bash
gostforge import-docx work.docx -o state.json   # импорт чужой работы
gostforge apply-fixes state.json -o state.json   # автофиксы T.08/T.10/T.14/...
gostforge ui                                     # визуальная правка
gostforge generate state.json -o final.docx     # финальная сборка
```

## 3.5 Отключение проверок для разделов

Титульный лист, реферат, приложения часто оформляются по своим правилам и
не должны проверяться стандартными правилами ГОСТа. Для этого:

**В builder API:**
```python
work("X", year=2026) \
    .section("Титульный лист")
        .paragraph("...")
        .skip_all_checks()                   # отключить все проверки
    .section("Приложение А")
        .paragraph("...")
        .skip_checks("H.01", "T.04", "T.05") # только конкретные коды
```

**В UI:** в редакторе раздела — раскрывающаяся панель «Нормоконтроль
раздела». Чекбокс «Не проверять этот раздел нормоконтролем» = отключить
всё. Иначе multi-select-ы по 15 категориям (F-Поля, T-Типографика,
H-Заголовки, S-Структура, R-Источники, B-Библиография, I-Рисунки/
таблицы, L-Списки, C-Перекрёстные ссылки, M-Формулы, A-Сокращения,
K-Колонтитулы, V-Объём, P-Приложения, X-Прочее).

**В JSON-state:**
```json
{"heading": "Титульный лист", "blocks": [], "disabled_checks": ["*"]}
{"heading": "Приложение А", "blocks": [], "disabled_checks": ["H.01", "T.04"]}
```

Реализация: `LogicalSection.disabled_checks: list[str]` (default `[]`).
Валидатор `engine.validate()` после прогона проверок отфильтровывает
violations, у которых `location` содержит id раздела с этим кодом
в disabled_checks. Глобальные нарушения (V.01 — объём,
F.01 — поля) **не** фильтруются — они относятся к документу в целом.

## 3.6 Подразделы 3-го уровня

`SectionBuilder.subsection()` рекурсивен — каждый вызов повышает level на 1.
Максимально поддерживается 3 уровня в UI и 9 в OOXML.

```python
sec = builder.section("Глава 1")             # level=1
sub = sec.subsection("1.1 Подраздел")        # level=2
pt  = sub.subsection("1.1.1 Пункт")          # level=3
pt.paragraph("Текст пункта.")
```

В UI: внутри expander «Подразделы» каждый подраздел имеет своих
«Пункты» (3-й уровень). Глубже не разворачивается, чтобы не перегружать
визуально.

При импорте чужой работы парсер кладёт LogicalSection плоско —
`document_to_state` реконструирует иерархию по level через
`_reconstruct_section_hierarchy`.

## 4. Визуальный редактор (Streamlit-WebApp)

Помимо CLI и программного API, доступен интерактивный визуальный
редактор. Запускается той же командой, что и режим нормоконтроля:

```bash
gostforge ui
```

В верхнем переключателе режима выберите **«Конструктор»**. Там же
доступны режимы **«Нормоконтроль»** (проверка существующего .docx),
**«История»** (все прошлые submission-ы из локальной БД с лентой
обсуждения руководитель↔студент) и **«Документация»** (встроенный
просмотр всех руководств `docs/*.md` без перехода на GitHub). WebApp
открывается в любом современном браузере, включая мобильный.

### Что умеет редактор

**Sidebar:**
- **Метаданные**: название работы, автор, научный руководитель,
  организация, год, вид работы, профиль ГОСТ.
- **Настройки стилей** (раскрывающаяся секция): переопределения
  параметров профиля для текущего документа без правки YAML — поля
  страницы (мм), шрифт/кегль/межстрочный/первый отступ/интервалы
  между абзацами, ВЕРХНИЙ-регистр и цвет/spacing заголовков, символ
  маркера и шаблон нумерации списков, стиль рамок таблиц.
- **Шаблоны работы**: кнопка «Загрузить шаблон» наполняет рабочее
  состояние скелетом курсовой / бакалаврской ВКР / отчёта о НИР.
- **Шаблоны разделов**: выпадающий список с готовыми разделами —
  Введение, Заключение, Реферат, Содержание, Список источников
  (с `is_bibliography=True`), Приложение А (с `disabled_checks=['*']`),
  Глава с подразделом 1.1.
- **Массовые операции** (bulk):
  - удалить пустые параграфы (рекурсивно по разделам и подразделам);
  - заголовки в Title Case (с поддержкой русского и цифр);
  - сбросить все skip-checks;
  - **авто-нумерация** — '1 Анализ', '1.1 Подраздел', '1.1.1 Пункт',
    с снятием нумерации со структурных разделов (Введение, Заключение,
    Содержание, Приложение, Список источников и т. д.).
- **Сохранение JSON** + **загрузка JSON** для перерывов в работе.
- **Загрузка готовой .docx в конструктор** — uploader `Загрузить .docx
  в конструктор`. Файл парсится через `parse_docx → document_to_state`,
  сразу прогоняется нормоконтроль и показывается сводка с кнопкой
  «Применить автофиксы».
- **Undo / Redo** на 50 шагов.

**Main-область (между sidebar и редактором раздела):**
- **Сводка импорта** — после `Загрузить .docx`: total + by_severity +
  топ-10 кодов нарушений + кнопка «Применить автофиксы», если для
  найденных кодов есть зарегистрированные фиксеры.
- **Комментарии рецензента** — если в импортированном .docx были
  комментарии Word (`word/comments.xml`), они показываются здесь
  с автором, датой и текстом.
- **Прогресс работы** — раскрывающийся expander со счётчиками
  заполнённых разделов / параграфов / таблиц / рисунков / источников /
  формул / элементов списков / слов / знаков и progress-bar.
- **Live-нормоконтроль** — постоянная сводка нарушений текущего state
  (обновляется при каждом изменении). Показывает total + by_severity +
  топ-5 кодов.

**Дерево разделов:**
- **Поиск** — text_input «Поиск по разделам» фильтрует дерево по
  case-insensitive substring; совпадения ищутся в заголовках всех
  уровней, текстах параграфов (включая rich-runs), подписях таблиц
  и рисунков, headers+cells таблиц, элементах списков, ссылках.
  Под кнопкой подсвечивается совпадение через `<mark>`.
- **6 кнопок на раздел**: маркер активности, заголовок (выбор),
  ↑/↓ (перемещение), ⎘ (дублирование с пометкой «(копия)»),
  ✕ (удалить).

**Редактор раздела:**
- Заголовок раздела (text_input).
- **Перемещение** через выпадающий список «Новая позиция».
- **Нормоконтроль раздела** (expander): чекбокс «Не проверять» и
  multi-select по 15 категориям кодов.
- Блоки: параграф (rich-runs), таблица (headers + rows + caption),
  рисунок (с загрузкой PNG/JPG), список (нумерованный / маркированный),
  формула (LaTeX).
- **Управление блоками** (кнопки у каждого блока): **↑ / ↓** —
  перемещение внутри раздела, **⎘** — дублирование (глубокая копия
  сразу после оригинала), **Удалить блок**. Те же кнопки работают и
  для блоков подразделов.
- **Перенос блока в другой раздел** — панель «Переместить блок в
  другой раздел»: выбор блока + целевого раздела (текущий и
  библиография исключены), блок уходит в конец выбранного раздела.
- **Подразделы 2 уровня** + **пункты 3 уровня** в раскрывающихся
  expander-ах.
- **Список источников**: специальный режим для `is_bibliography=True`
  разделов, где каждая строка — отдельная запись по ГОСТ Р 7.0.100-2018.

**Внизу страницы:**
- **«Сгенерировать .docx»** — сборка через `WorkBuilder`, скачивание
  файла, сводка нарушений после сборки.
- **«Сгенерировать + превью PDF»** — то же + конвертация в PDF через
  LibreOffice headless и inline-просмотр в `<iframe>` с base64
  data URL.

**Авто-сохранение** в `~/.gostforge/autosave/last-session.json` —
periodically (не чаще раза в 30с). При старте UI, если обнаружен
свежий (<24 часов) файл и текущий state ещё дефолтный, показывается
баннер «Восстановить / Игнорировать».

### Архитектура

Единственный источник истины — `st.session_state.builder_state`. Любое
действие пользователя мутирует state, после чего Streamlit
перерисовывает страницу. Внутренние функции
`_build_document_from_state(state) -> bytes` и
`_document_to_state(document) -> dict` лежат в
`src/gostforge/web/builder_editor.py` и покрыты unit-тестами без
поднятия Streamlit-сессии.

### 4.1 Пословное редактирование (Фаза 2.5)

Параграф редактируется на уровне **inline-элементов**, а не одной
строкой. Каждый run — отдельная редактируемая строка с собственными
управляющими элементами (B/I/U/sup/sub для текста, select для
ссылок/цитат, LaTeX-input для inline-формул).

Поддерживаемые типы inline-элементов:

| Тип | UI-редактор | На экспорте → |
| --- | --- | --- |
| **TextRun** | `text_area` + чекбоксы `B`, `I`, `U`, `x²`, `x₂` | `<w:r>` с `<w:rPr>` |
| **InlineFormula** | input LaTeX + live-preview через `st.latex` | `<w:r><m:oMath>...</m:oMath></w:r>` (inline OMML) |
| **CrossRef** | `selectbox` со списком всех figure/table/numbered formula + опц. prefix | `<w:fldSimple w:instr=" REF target_id \h "/>` с опц. предшествующим run |
| **Citation** | `selectbox` с записями bibliography + опц. pages | текстовый run «[N]» / «[N, с. P]» |

Под панелью run-ов — 4 кнопки: `+ Текст`, `+ Формула`, `+ Ссылка`,
`+ Цитата`. Для каждого run-а — стрелки `↑` / `↓` для перемещения и
`×` для удаления.

#### Схема state

Параграф в `builder_state.sections[i].blocks[j]`:

```json
{
  "kind": "paragraph",
  "runs": [
    {"kind": "text", "text": "Энергия ", "bold": false},
    {"kind": "formula", "latex": "E = h\\nu"},
    {"kind": "text", "text": " описана в "},
    {"kind": "citation", "source_id": "bib-1", "pages": "42",
     "template": "[{n}, с. {pages}]"},
    {"kind": "text", "text": "."}
  ]
}
```

Legacy формат Фазы 2 `{"kind": "paragraph", "text": "..."}` тоже
поддерживается — при загрузке JSON автоматически вызывается
`_normalize_paragraph_state`, который конвертирует `text` → один
TextRun-dict.

Для CrossRef и Citation в state используются proxy-id:

- **`fig-N` / `tbl-N` / `formula-N`** — стабильные идентификаторы по
  порядковому номеру в работе. Совпадают с теми id'ами, что генерирует
  `WorkBuilder` при `build()` — никакого перевода не требуется.
- **`bib-N`** — порядковый номер записи в библиографии. На сборке
  документа `_resolve_citation_proxies` заменяет `bib-N` на реальный
  `BibliographyEntry.id`, назначенный билдером. Невалидные индексы
  (например, `bib-99` при 5 записях) оставляются как есть — экспортёр
  выдаст «[?]», что для проверки R.04 является сигналом ошибки.

#### Undo / Redo

Кольцевой буфер из 50 snapshot-ов в `st.session_state.builder_history`,
позиция — в `builder_history_cursor`. В sidebar — кнопки
`⟲ Отменить` / `⟳ Повторить` и индикатор «Snapshot N из M».

Snapshot делается лениво: в начале каждого rerun
`_auto_snapshot_if_changed` сравнивает текущий state с history[cursor].
Если они отличаются — пушится новый snapshot. После undo state
становится равен history[cursor], поэтому auto-snapshot не срабатывает
— undo/redo остаются полностью обратимыми.

После undo любая новая мутация отрезает «redo-будущее» (классический
branch-and-truncate, как в текстовых редакторах).

#### Auto-save

State периодически (не чаще раза в 30 секунд) сериализуется в
`~/.gostforge/autosave/last-session.json`. При старте UI, если
обнаружен свежий (<24 часов) autosave-файл и текущий state ещё
дефолтный, показывается баннер «Восстановить / Игнорировать».

Ошибки IO (нет места, нет прав) логируются и проглатываются — UI
продолжает работать без autosave.

#### Программный API

Для пословного контента доступен низкоуровневый builder-метод:

```python
from gostforge.builder import work
from gostforge.model import (
    Citation,
    CrossRef,
    InlineFormula,
    TextRun,
)

(
    work("Курсовая", year=2026)
    .section("Введение")
        .rich_paragraph([
            TextRun(text="Энергия фотона "),
            InlineFormula(latex="E = h\\nu"),
            TextRun(text=" описана в "),
            Citation(source_id="planck-1900", pages="42"),
            TextRun(text=" (см. "),
            CrossRef(target_id="fig-1", prefix=""),
            TextRun(text=")."),
        ])
    .save("paper.docx")
)
```

`.paragraph(text, bold=, italic=)` остаётся обёрткой над
`.rich_paragraph([TextRun(...)])` — для случаев, когда форматирование
сквозное по всему абзацу.
